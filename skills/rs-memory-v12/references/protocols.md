## Section: QUALITY — Code Quality Framework

Every tool delivered to RS must pass ALL quality gates:

### Mandatory Quality Gates

| Gate | Standard | Verification |
|------|----------|-------------|
| **Error Handling** | try/except on ALL external calls | Every network, file, API call wrapped |
| **Logging** | Configurable logging system | logging module with levels (DEBUG/INFO/WARNING/ERROR) |
| **Modularity** | Independent functions/classes | No god functions, single responsibility |
| **Configuration** | Config file or CLI args | No hardcoded values except defaults |
| **Progress** | Visual indicators for long ops | Progress bars, spinners, percentage |
| **Cleanup** | Resource cleanup on exit | Context managers, finally blocks, signal handlers |
| **Cross-Platform** | Kali + Termux compatible | Path handling, dependency checks |
| **Signals** | SIGINT/SIGTERM handling | Clean shutdown on Ctrl+C |
| **Documentation** | Docstrings + usage guide | Every function documented |
| **Testing** | Testable design | Functions return values, not just print |
| **Input Validation** | All user input validated | Type checking, range checking, sanitization |
| **Rate Limiting** | Configurable delays | Avoid triggering WAF/IDS when scanning |
| **Output Formats** | Multiple output options | JSON, CSV, text, HTML at minimum |
| **Threading/Async** | Concurrent operations | threading or asyncio for I/O-bound tasks |
| **Color Output** | Terminal colors for readability | colorama or ANSI codes |

### Architecture Patterns

**Single-File Tool Pattern:**
```
#!/usr/bin/env python3
"""Tool Name - Description"""

# Imports
# Constants / Configuration
# Utility Functions
# Core Classes/Functions
# Output/Reporting Functions
# Main / CLI
# Signal Handlers
# Entry Point
```

**Multi-File Suite Pattern:**
```
tool-name/
├── main.py          # Entry point + CLI
├── config.py        # Configuration management
├── core.py          # Core logic
├── utils.py         # Utility functions
├── output.py        # Output formatting + reporting
├── requirements.txt # Dependencies
└── README.md        # Usage documentation
```

| "red team" | Full engagement | Chain from Playbook 6 |

### Chain Architecture
```
Tool_A (output: JSON) → Tool_B (input: JSON from A, output: JSON) → Tool_C (input: JSON from B)
```
- Each tool in the chain reads the previous tool's output
- JSON is the standard interchange format between chain links
- The chain has a unified entry point and unified report at the end
- Each link can also run standalone

### Chain Delivery
When delivering a chain:
1. Deliver the COMPLETE chain as a single integrated tool
2. Each step is a function/module within the tool
3. Menu allows running individual steps or the full chain
4. Results from each step are saved and passed to the next
5. Final report summarizes the entire chain's findings
